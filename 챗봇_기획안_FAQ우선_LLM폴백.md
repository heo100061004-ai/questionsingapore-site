# Question Singapore 챗봇 기획안 (FAQ 1차 + LLM 2차 Fallback)

## 1. 문서 목적
이 문서는 Question Singapore 사이트에 적용할 챗봇의 제품 방향, 운영 정책, 비용 전략, 데이터 구조, 구현 단계, 리스크 대응안을 정의한다.
핵심 목표는 일반 정보 응답 제공과 문의 전환 최적화이며, 비용은 낮게 유지한다.

## 2. 배경 및 문제 정의
- 사용자는 고용/부동산/리로케이션 관련 일반 정보를 빠르게 알고 싶어한다.
- 모든 질문을 사람이 바로 응대하기에는 운영 부담이 크다.
- LLM만으로 전체 응답을 처리하면 비용이 빠르게 증가할 수 있다.
- 전문/개별 판단이 필요한 문의는 궁극적으로 상담 접수로 연결되어야 한다.

## 3. 제품 목표
### 3.1 비즈니스 목표
- 챗봇을 통해 초기 질문 이탈을 줄인다.
- 사용자의 기본 궁금증을 즉시 해소한다.
- 최종적으로 사이트 문의 접수(이메일/WhatsApp) 전환율을 높인다.

### 3.2 운영 목표
- LLM 호출을 최소화해 저비용 구조를 유지한다.
- 답변 품질과 안전성(과도한 확정적 자문 방지)을 통제한다.
- FAQ를 운영 데이터 기반으로 점진 개선한다.

## 4. 핵심 방향성
1. 메인 페이지에 FAQ 목록을 크게 노출하지 않는다.
2. 챗봇이 내부 FAQ 문서를 우선 검색해 답변한다.
3. FAQ에서 답을 찾지 못할 때만 LLM fallback 호출한다.
4. 전문 판단이 필요한 케이스는 반드시 문의 접수로 유도한다.

## 5. 사용자 플로우
1. 사용자가 챗봇에 질문 입력
2. 카테고리 분류(고용/부동산/리로케이션/기타)
3. FAQ 유사도 검색
4. 유사도 임계치 이상: FAQ 기반 답변 반환
5. 유사도 임계치 미만: LLM fallback 호출
6. 답변 말미에 문의 CTA 고정 노출
7. 사용자가 원하면 문의 폼으로 이동 + 대화 요약 자동 입력

## 6. 범위 정의
### 6.1 인스코프
- 일반/제너럴 정보 안내
- 체크리스트, 준비 순서, 기본 개념 설명
- 문의 접수 안내 및 링크 유도

### 6.2 아웃오브스코프
- 법률/세무/이민의 확정적 자문
- 개별 계약/분쟁에 대한 단정적 판단
- 특정 사례의 책임 있는 전문 판정

## 7. 답변 정책 (가드레일)
- 모든 답변은 "일반 정보 제공"임을 전제로 한다.
- 규제/계약/비자 조건은 최신 정책 변경 가능성을 명시한다.
- 확정적 표현(예: 반드시 승인된다, 무조건 가능하다)을 금지한다.
- 위험도 높은 문의는 즉시 전문가 문의를 권고한다.

## 8. 데이터/문서 구조
## 8.1 FAQ 저장 구조 (제안)
- 경로 예시: config/faq.json
- 기본 필드
  - id
  - category (employment | real_estate | relocation)
  - question
  - answer
  - keywords[]
  - language (ko | en | zh)
  - updatedAt
  - sourceType (manual | approved_suggestion)
  - riskLevel (low | medium | high)

## 8.2 FAQ 제안 큐 구조 (제안)
- 경로 예시: config/faq_suggestions.json
- 기본 필드
  - suggestionId
  - fromQuestionId
  - category
  - proposedQuestion
  - proposedAnswer
  - similarityScore
  - status (pending | approved | rejected)
  - reviewedBy
  - reviewedAt

## 9. FAQ 자동 업데이트 전략
완전 자동 반영이 아니라 "자동 제안 + 관리자 승인" 방식을 채택한다.

1. 답변 완료 문의 발생
2. 시스템이 FAQ와 유사도 비교
3. 신규/보강 후보 제안 생성
4. 안전성 필터 수행(개인정보/과도 확정 표현/리스크)
5. 관리자 큐에서 승인/수정/반려
6. 승인된 항목만 FAQ 본문 반영

## 10. 비용 전략 (저비용 설계)
1. FAQ 1차 응답을 기본값으로 사용
2. FAQ 미매칭 시에만 LLM 호출
3. 웹 검색 호출은 고난도 질문에 제한 사용
4. 세션당 LLM 호출 횟수 제한
5. 답변 길이 제한
6. 동일/유사 질문 캐시
7. 일일 예산 상한 설정

## 11. KPI
- 챗봇 시작률
- FAQ 매칭률 (LLM 미호출 비중)
- 챗봇 → 문의 폼 전환율
- 카테고리별 전환율
- 평균 대화 턴 수
- 문의당 운영비(추정)

## 12. 구현 단계 (MVP 기준)
### Phase 1: FAQ 기반 챗봇
- FAQ 데이터 스키마 확정
- 클라이언트 챗 UI 기본 구현
- FAQ 유사도 매칭 로직 구현
- 문의 CTA 연결

### Phase 2: LLM fallback
- FAQ 미매칭 시만 LLM 호출
- 응답 가드레일/금칙어/문구 정책 적용
- 비용 제한 장치 적용

### Phase 3: FAQ 제안/승인 워크플로
- 답변 완료 건 기반 FAQ 후보 생성
- 관리자 승인 큐 UI
- 승인 시 FAQ 반영 자동화

## 13. 예상 일정
- Phase 1: 0.5~1일
- Phase 2: 0.5~1일
- Phase 3: 1~2일
- 총: 약 2~4일 (MVP~운영형 초입)

## 14. 리스크 및 대응
1. 오답/환각 리스크
- 대응: FAQ 우선, LLM fallback 제한, 안전 문구 강제

2. 비용 급증 리스크
- 대응: 호출 제한, 길이 제한, 캐시, 예산 캡

3. 법적/규제 민감 답변 리스크
- 대응: 확정 자문 금지, 고위험 문의 즉시 전문가 유도

## 15. 운영 가이드
- FAQ는 주 1회 점검/월 1회 대규모 업데이트
- FAQ 제안 큐는 관리자 승인 후 반영
- 반영 로그(누가/언제/무엇을 변경했는지) 유지

## 16. 결론
본 기획은 "비용 최소화 + 전환 최대화"를 동시에 달성하는 구조다.
FAQ 기반 1차 응답으로 운영비를 줄이고, LLM fallback으로 답변 범위를 보완하며, 최종적으로는 문의 접수로 연결해 사업 목표를 달성한다.
